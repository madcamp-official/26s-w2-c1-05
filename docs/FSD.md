# 여보세요 — 콜포비아 실전 트레이닝 게임 기능 명세서

---

## 1. 프로젝트 개요

### 1.1 한 줄 정의
전화 공포(콜포비아)를 겪는 MZ세대가 "훈련"이 아니라 "게임"으로 전화 실력을 키우는 크로스 플랫폼 앱. AI 보스와의 전화 보스전(싱글)과 유저 간 실시간 전화 배틀(멀티)을 코어 루프로 하며, 통화 중 실시간 게임 연출(모멘텀 게이지·이벤트·코치)과 판 종료 후 LLM 피드백 리포트를 제공한다.

### 1.2 문제 정의
- 콜포비아는 공인된 현상이지만 국내 솔루션은 전화 대행·AI 대신 받기 같은 회피형만 존재
- 해외에는 murmur, KallyConfidence 등 훈련형 앱이 등장해 카테고리는 검증됨. 그러나 전부 1인용 훈련 앱에 장식적 게임 요소를 얹은 수준
- 한국어 + 한국 생활 시나리오 + 게임 본체 + 실시간 멀티플레이 조합은 공백

### 1.3 포지셔닝
- "훈련 앱에 게임을 뿌린 것"이 아니라 "게임인데 실력이 남는 것"
- 차별화 3축: ① 한국어 화법 로컬라이제이션(존댓말 층위, 돌려 말하기) ② 보스전 컬렉션이라는 게임 본체 구조 ③ 경쟁자 전무한 실시간 유저 대전
- 시장 검토 프레임: "세계 최초"가 아니라 "해외 검증된 카테고리 + 한국어 공백 + 게임화·멀티플레이 차별화"

### 1.4 과제 요건 충족
| 요건 | 충족 방식 |
|---|---|
| LLM 활용 | 보스 페르소나 대화, 인크리멘탈/최종 심판, 피드백 리포트, 시나리오 생성 |
| 실시간 요소 | 음성 스트리밍 대화, 실시간 매칭 큐, 유저 간 통화, 실시간 모멘텀 게이지 |
| 다중 사용자 | 전화 배틀(1:1 매칭), 크로스 폼팩터 대전, 리더보드 |
| 크로스 플랫폼 | Flutter 단일 코드베이스 → Android 앱 + Windows 데스크톱 앱, 두 폼팩터 간 대전 |
| 데모 임팩트 | 심사위원 현장 보스전 도전 + 모바일↔데스크톱 배틀 시연 + 데스크톱 관전 화면 |

## 2. 플랫폼 구성

### 2.1 크로스 플랫폼 전략
- **프레임워크**: Flutter 단일 코드베이스 (팀의 React 경험 전이 용이, Dart)
- **타겟**: Android 앱 + Windows 데스크톱 앱. iOS·웹 없음 (과제 요건: 웹 기반 프레임워크 대신 다른 프레임워크 시도)
- **두 폼팩터는 동등한 완전 게임 클라이언트**: 데스크톱 유저도 앱 하나로 보스전·배틀·통화 전부 수행. 데스크톱은 컴패니언/관전 전용이 아님
- **크로스 폼팩터 연동**: 배틀에서 모바일 유저 ↔ 데스크톱 유저가 매칭·대전 가능. 이것이 크로스 플랫폼 어필의 중심

### 2.2 폼팩터별 역할(레이아웃 차이일 뿐, 기능은 동등)
- **모바일(세로)**: "전화기" — 통화의 본체 경험, 세로 단일 흐름
- **데스크톱(가로)**: 넓은 화면 활용 — 대부분 화면은 네이티브 와이드 레이아웃(다열 그리드), 통화 화면은 중앙 폰 목업 + 양옆 게임 HUD, 관전·대시보드에서 추가 이점

## 3. 코어 게임 모드

### 3.1 [P0] 싱글플레이 — 보스전 컬렉션 "전설의 진상 도감"

#### 3.1.1 보스 도감
- 난이도별 보스 = 시나리오 + 페르소나 + 클리어 조건의 묶음. 8종 초기 라인업:

| # | 보스 | 시나리오 | 클리어 조건 |
|---|---|---|---|
| 1 | 무던한 치킨집 사장님 | 배달 주문 | 메뉴·주소·요청사항 전달 완료 |
| 2 | 바쁜 미용실 원장 | 예약 잡기 | 원하는 시간대 예약 확정 |
| 3 | 따발총 치과 접수원 | 진료 예약 + 보험 질문 | 예상 못 한 질문 2개 이상 처리 |
| 4 | 한숨 쉬는 공무원 | 서류 발급 문의 | 필요 서류 목록 정확히 확보 |
| 5 | 말 끊는 거래처 부장 | 업무 일정 조율 | 끊겨도 용건 3가지 모두 전달 |
| 6 | 예약 멋대로 바꾼 미용실 | 항의 + 재조정 | 사과 또는 보상 확보 |
| 7 | 반말하는 사장님 | 아르바이트 급여 문의 | 감정 동요 없이 지급일 확정 |
| 8 | 환불 불가 3연벙 상담원 (최종) | 환불 요구 | 환불 또는 동등 보상 확보 |

- 잠금/해금: 이전 난이도 클리어 시 해금. 클리어 시 도감 등록 + 통화 하이라이트 저장

#### 3.1.2 통화 플로우
1. 보스 선택 → 목표 브리핑(용건, 클리어 조건, 제한 시간)
2. "발신" → 전화 UI 전환(신호음 → 보스 응대)
3. Push-to-talk 턴제: 유저 발화(STT) → LLM 응답(스트리밍) → TTS 재생
4. 종료: 클리어 / 시간 초과 / 침묵 누적 / 끊기
5. 결과 + 피드백 리포트

#### 3.1.3 보스 행동 규칙 (LLM 프롬프트 요구사항)
- 페르소나 일관성: 말투·성격·인내심을 시스템 프롬프트에 고정, few-shot 예시 3개
- 응답 길이 강제: 1~2문장(스트리밍 첫 문장 TTS 시작을 위해)
- 매 판 랜덤 변수 2~3개 주입 → 리플레이성
- 난이도 파라미터: 응답 길이, 협조성, 돌발 질문 빈도, 말 끊기
- 금지: 인신공격, 실제 업체·인물 언급

#### 3.1.4 인내심 게이지 (인크리멘탈 심판의 싱글 적용)
- 통화 중 주기적으로 보스의 "인내심"을 시각화. 클리어 조건 진행(1/3→2/3)도 이벤트 팝업으로

### 3.2 [P1] 멀티플레이 — 실시간 "전화 배틀"

#### 3.2.1 매칭
- 매칭 큐 진입 → 1:1 매칭(초기 필터 없음, 크로스 폼팩터 허용) → 역할 랜덤 배정(민원인/상담원)
- 콜드스타트 폴백: 30초 내 매칭 실패 시 "AI 상담원과 배틀" 제안

#### 3.2.2 게임 룰 — 지배 전략("무조건 버티기") 제거
- **비밀 목표**: LLM이 각자에게 비공개 목표 전달 (민원인: "환불 받아내라" / 상담원: "환불 없이 만족도 3점 이상으로 종료")
- **히든 규정 카드**: 상담원 전용, 판마다 변동 (예: 소비자원 신고·녹취 고지·본사 항의 언급 시 접수 의무 발생)
- **과정 채점**: 결과가 아닌 심판의 과정 점수로 판정
  - 상담원 감점: 앵무새 반복, 근거 없는 거절, 고객 무시, 무례, 일방적 끊기
  - 상담원 가점: 대안 제시, 규정 근거 설명, 감정 진정
  - 민원인 감점: 논점 없는 화내기, 용건 불명확
  - 민원인 가점: 올바른 에스컬레이션 절차 발견, 논리적 지적
- 제한 시간 5분 (데모 3분 지원)

#### 3.2.3 통신 구조 (B-lite 우선)
- 음성: **B-lite** — 각자 STT 텍스트를 WebSocket으로 릴레이 → 상대 기기에서 TTS 재생. (P2: flutter_webrtc P2P 음성)
- 기록: 각 클라이언트 STT 결과 `{화자, 텍스트, 발화시작 타임스탬프}`를 서버로 전송, 서버가 상대 시각(통화 시작=0초) 기준 병합
- 서버 역할: 방 상태 관리, 타이머 동기화, 침묵 감지, 조기 종료 처리

## 4. 인크리멘탈 심판 (실시간 게임 층)

### 4.1 목적
"그냥 둘이 전화하는 것"에 격투 게임의 실시간 피드백 층을 얹어 게임 감각 부여. 통화를 막지 않는 비동기 사이드 채널.

### 4.2 동작
- 서버가 병합 트랜스크립트에 새 발화가 3~4개 쌓이거나 20초 경과 시 경량 심판(자체 서빙 모델) 호출
- 입력: 누적 상태(현재 게이지, 발동 이벤트 목록, 각자 목표) + 델타 대화
- 출력(JSON): `momentum_delta`, `event`(규정 발동/클린히트/페널티), `hint_A`/`hint_B`(각자 전용, 상대 비밀 누설 금지), `caster`(중계 코멘트)

### 4.3 산출 UI 요소
| 요소 | 위치 | 비고 |
|---|---|---|
| 모멘텀 게이지 | 통화 화면 상단(양측 공통) | 배틀=줄다리기, 싱글=보스 인내심. 버티기 전략이 실시간 처벌됨을 시각화 |
| 이벤트 팝업 | 통화 화면 오버레이 | "규정 카드 발동!", "논리 클린히트!" 등, 2초 자동 소멸 + 효과음 |
| AI 코치 귓속말 | 본인 전용(데스크톱=우측 패널 로그) | 화술 코칭. 상대 비밀·규정 누설 금지 |
| 실황 캐스터 | 관전 화면 전용 | 배틀 당사자 화면엔 미표시 |

### 4.4 판정 신뢰 장치
- 중간 게이지는 참고 지표. **최종 승패는 종료 후 전체 트랜스크립트 정밀 심판이 결정** → 중간 오판이 게임을 망치지 않고 역전 서사 유발

## 5. 음성 파이프라인 (폼팩터별 이원화)

### 5.1 실시간 STT/TTS
| 폼팩터 | STT (실시간) | TTS |
|---|---|---|
| Android | `speech_to_text` (OS 내장, interim 지원, 무료, 저지연) | `flutter_tts` (OS 내장) |
| 데스크톱(Windows) | `record`로 오디오 캡처 → WebSocket → 서버 faster-whisper | `flutter_tts` (Windows SAPI) |

- STT를 `SttEngine` 인터페이스로 추상화(onInterim/onFinal), 플랫폼별 구현체 교체
- 데스크톱 STT 실패 시 텍스트 입력 폴백 필수

### 5.2 심판 정제 트랙 (플랫폼 무관)
- 실시간 자막은 온디바이스(빠름). 최종 심판·리포트용 트랜스크립트는 통화 오디오를 발화 단위로 서버 faster-whisper에 병렬 전송해 정제(정확)
- 효과: 배틀에서 기기별 STT 편차로 판정이 갈리는 문제 해소, 플랫폼 간 채점 균일성 확보

### 5.3 지연 비대칭 대응
- 폼팩터별 STT 지연 차(온디바이스 ~0.3초 vs Whisper 0.8~2초)는 대화엔 영향 없고 기록·채점에만 영향
- **채점은 발화 시작 타임스탬프(통화 시작=0초) 기준** → latency-agnostic
- 인크리멘탈 심판 트리거를 "양측 발화 확정 수신 후"로, 미도착 발화는 다음 턴 이월
- 제한 시간 경계: 종료 후 3초 "지각 트랜스크립트 유예 창", 타임스탬프가 시간 내면 인정

### 5.4 레이턴시 최적화 (싱글 통화)
- Push-to-talk로 무음 감지 대기 제거
- LLM 스트리밍 → 첫 문장 완성 즉시 TTS 시작, 보스 응답 1~2문장 강제
- 응답 대기 중 "잠시만요…" 타이핑 효과음으로 딜레이를 리얼리즘으로 흡수
- 목표 체감 지연: 1~1.5초

## 6. LLM 연동 설계 (하이브리드 배분)

### 6.1 서빙 인프라
- 자체 ML 서버: RTX 3090 20GB, vCPU 40 / RAM 50GB / DISK 100GB
- 서빙: vLLM (OpenAI 호환 API, 스트리밍, guided decoding으로 JSON 스키마 강제)
- 모델: **Qwen3-14B AWQ** (faster-whisper 동거 위해 30B-A3B 대신 14B 확정. Whisper small ~1-2GB + 14B ~9-10GB로 20GB 내 여유)

### 6.2 태스크별 모델 배분
| 태스크 | 모델 | 빈도 | 이유 |
|---|---|---|---|
| 보스 대화 턴 | 자체 서빙(Qwen3-14B) | 고 | RPD 무제한, 저지연 |
| 인크리멘탈 심판 | 자체 서빙 + guided JSON | 고 | 판당 15회+를 무료로, 스키마 강제 |
| 최종 심판 + 리포트 | Gemini 2.5 Flash | 저(판당 2회) | 판정 품질=신뢰, 저빈도라 무료 티어로 충분 |
| 시나리오/변수 생성 | Gemini 2.5 Flash | 저(판당 1회) | 창의성·다양성 필요 |

- LLM 클라이언트를 인터페이스로 추상화(둘 다 OpenAI 호환 형식) → 서버 다운 시 환경변수로 전량 Gemini 폴백(데모 보험)

### 6.3 프롬프트 자산
| 프롬프트 | 역할 | 출력 | 실행 |
|---|---|---|---|
| 보스 페르소나 | 캐릭터 응대 | 텍스트(짧은 발화) | 자체 서빙 |
| 인크리멘탈 심판 | 델타 채점·게이지·힌트·캐스터 | JSON | 자체 서빙 |
| 최종 심판 | 승패·루브릭·근거 대사 인용 | JSON | Gemini |
| 리포트 생성기 | 피드백 리포트 | JSON | Gemini |
| 시나리오/변수 생성기 | 배틀 시나리오·목표·규정, 보스 변수 | JSON | Gemini |
- 공통 안전 규칙: 인신공격 금지, 실존 업체·인물 금지, 힌트의 상대 비밀 누설 금지

### 6.4 파인튜닝 (조건부)
- 대상은 "보스 대화 턴" 하나(말투·형식 고정). 심판·리포트·시나리오는 대상 아님
- 방법: Gemini로 보스 페르소나 모의 통화 대량 생성 → Qwen3-14B에 QLoRA 증류(Unsloth/LLaMA-Factory). vLLM LoRA 동적 로딩으로 보스별 어댑터 가능
- 판단 기준: 프롬프트로 "말투 붕괴"가 안 잡히고 일정 여유 있을 때만. Day 1부터 Gemini 응답 로깅해 데이터 준비
- 리스크: 어설픈 튜닝은 catastrophic forgetting으로 JSON 능력 손상 → 좁게 적용

### 6.5 자원 배분 (3090 20GB)
- vLLM: Qwen3-14B AWQ + faster-whisper small 동거. `--max-model-len 4096`으로 KV 캐시 확보
- vCPU 40 / RAM 50GB에 게임 서버(백엔드) + STT 후처리 동거 가능

## 8. 기술 스택 요약

- 앱: Flutter (Android + Windows 단일 코드베이스)
- STT: Android `speech_to_text`, Desktop `record`→faster-whisper; TTS: `flutter_tts`

- LLM: vLLM(Qwen3-14B AWQ) + Gemini 2.5 Flash; STT 정제: faster-whisper @ 3090
- DB: 유저, 보스 클리어, 전적, 트랜스크립트

- 인터페이스 Day 1 정의: `SttEngine`, `TtsEngine`, `AudioRecorder`, `LlmClient`. 구현은 Android+Windows만
- UI는 처음부터 mobile/desktop 두 폼팩터 반응형
- 조건부 import로 구현체 교체 → 향후 웹 확장은 "구현체 추가"라는 좁은 태스크로 격리

## 11. 데모 시나리오 (발표용)

1. **싱글 데모(2분)**: 심사위원에게 기기 건네 "환불 상담원 보스전" → 현장 긴장+웃음 → 리포트 공개
2. **크로스 폼팩터 배틀(2분)**: 팀원 A(모바일) vs 팀원 B(데스크톱) 매칭 → 데스크톱 관전 화면을 프로젝터에 → 모멘텀 게이지·캐스터로 중계 → 판정 + 비밀 목표 공개
3. **엔지니어링 스토리**: 태스크별 모델 계층화(자체 서빙+Gemini), 폼팩터별 STT 추상화, 타임스탬프 채점으로 지연 공정성 확보
4. **선제 방어**: "버티면 이기는 것 아니냐 → 과정 채점·히든 규정으로 지배 전략 제거"
